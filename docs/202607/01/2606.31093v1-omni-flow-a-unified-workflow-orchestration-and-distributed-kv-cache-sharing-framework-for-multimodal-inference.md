## 摘要
随着大语言模型（LLM）推理从纯文本范式向多模态范式演进，推理系统面临三大挑战：（1）多模态工作流的灵活编排，其中异构计算单元呈现复杂依赖与并发控制；（2）跨进程与节点的高效海量中间数据传输，张量在异构角色间高速流动；（3）跨角色的KV缓存与模型权重高效共享以消除冗余GPU内存。现有方案独立部署LLM与扩散模型，缺乏面向多模态流水线的系统级抽象；这导致编排逻辑分散，传输路径与特定模型紧耦合，且集成新模型成本高昂。为应对这些挑战，我们提出Omni-Flow，一个通过三层抽象实现多模态推理的分布式调度框架。控制流层通过Python DSL定义工作流，将异构单元编排为统一数据流图，支持静态DAG与动态路由，内置服务发现与多种负载均衡策略。数据流层提供超越预填充/解码分离的分布式KV缓存抽象，统一分配并支持通过零拷贝、低延迟通道在三层分页存储层次（GPU/CPU/SSD）间进行直接跨角色传输。计算流层支持多轮对话中KV复用的复杂多模态前缀匹配，并通过统一SGLang接口接管KV缓存与采样逻辑，使扩散模型能在统一并行语义下直接复用LLM前向路径。我们证明Omni-Flow能以一致编程模型支持多种异构场景，包括全模态对话（LongCat-Next）与复杂图像生成流水线（HunyuanImage-3）。

## Abstract
As large language model (LLM) inference evolves from text-only to multimodal paradigms, inference systems face three challenges: (1) flexible orchestration of multimodal workflows, where heterogeneous computing units exhibit complex dependencies and concurrent control; (2) efficient transmission of massive intermediate data across processes and nodes, with tensors flowing at high speed among heterogeneous roles; and (3) efficient sharing of KV caches and model weights across roles to eliminate redundant GPU memory. Existing solutions deploy LLMs and diffusion models independently, lacking a system-level abstraction for multimodal pipelines; this scatters orchestration logic, tightly couples transmission paths to specific models, and incurs high cost to integrate new models. To address these challenges, we present Omni-Flow, a distributed scheduling framework for multimodal inference through a three-layer abstraction. The Control Flow layer defines workflows via a Python DSL, orchestrating heterogeneous units into a unified dataflow graph that supports static DAGs and dynamic routing, with built-in service discovery and diverse load-balancing strategies. The Data Flow layer provides a distributed KV cache abstraction beyond prefill/decode separation, unifying allocation and enabling direct cross-role transmission across a three-tier paged storage hierarchy (GPU/CPU/SSD) over zero-copy, low-latency channels. The Compute Flow layer supports complex multimodal prefix matching for KV reuse across multi-turn dialogues, and takes over KV cache and sampling logic via a unified SGLang interface, letting diffusion models directly reuse the LLM forward path under unified parallel semantics. We demonstrate that Omni-Flow supports diverse heterogeneous scenarios with a consistent programming model, including omni-modal dialogue (LongCat-Next) and complex image generation pipelines (HunyuanImage-3).

## 2. 方法论：核心思想、关键技术细节、算法流程

### 核心思想
Omni-Flow通过**三层抽象**解耦工作流编排、数据传输和计算执行，实现统一的异构多模态推理：

- **控制流（Control Flow）**：用声明式Python DSL定义工作流，将异构计算节点（LLM、扩散模型、编码器等）组织为数据流图，支持静态DAG和动态分支、循环等高级语义。
- **数据流（Data Flow）**：提供统一的分布式数据平面，管理KV缓存和中间张量的生命周期，支持跨角色、跨机器、跨存储层级（GPU/CPU/SSD）的零拷贝共享。
- **计算流（Compute Flow）**：基于SGLang深度定制，统一LLM推理内核，实现多模态前缀匹配、KV缓存接管，使扩散模型复用LLM执行路径。

### 关键技术细节

#### 控制流层
- **图定义**：Role、Placeholder、GlobalParameters、RoleRef等核心组件；输入采用**OR-AND语义**（二维列表）。
- **三种依赖关系处理**：
  - **钻石依赖（Diamond）**：通过Bucket模型和`activation id`（基于公共源节点的帧序号哈希）实现多路径汇聚同步。
  - **序列依赖（Sequence）**：在SLB（SessionLoadBalance）节点上维护期望序号，实现顺序保证，其他节点无排队。
  - **多流连接（Multi-Stream Join）**：通过`activate_by_count`声明，强制将多路径归入同一Bucket，按字段计数触发。
- **静态计划（StaticPlan）**：编译时预计算边表、控制边、钻石根、关键槽位，运行时仅查表。
- **动态图执行**：节点调度器（NodeController）管理生命周期，支持不可达信号传播、任务完成检测（ActivityTracker）、跨图调用序列化（GCScheduler）等。
- **服务发现与负载均衡**：基于Redis的ServiceRegistry，支持RR、SLB策略。

#### 数据流层
- **全局参数池（Global Params Pool）**：统一管理KV缓存在L1/L2/L3存储层的分配、迁移和回收，支持RDMA跨节点拉取。
  - **物理分页层**：按层/head灵活组合。
  - **多层级槽引擎**：TinyLFU+四层LRU驱逐策略；引用计数保证安全；反刍合并上传。
  - **分布式元数据协调**：Redis Lua脚本原子操作，布局ID+预合并算法，复制品选择优先覆盖多页节点，三层保护驱逐。
- **临时缓冲区池（Tmp Buffer Pool）**：预分配slab，引用计数+LRU双轨复用，支持跨节点按需拉取（pull-on-demand）。
- **权重共享**：通过monkey-patch PyTorch工厂函数，将所有权重分配为IPC共享张量，多角色复用同一物理内存。
- **KV缓存共享**：co-located角色共享统一缓存池，通过`kv_pool_barrier`避免资源高估。

#### 计算流层
- **前缀匹配**：分为**全量匹配**（适合无复杂模态输出的场景）和**增量匹配**（支持引用历史模态输出）。共享**链式分页哈希**机制，将前缀压缩为扁平哈希串。
- **接口抽象**：接SGLang，通过统一插件接管内存和采样逻辑，输入输出由配置键列表驱动。
- **扩散模型复用LLM路径**：DiT模块继承LLM的KV和权重管理基类，将去噪步骤组织为序列执行标准分页注意力，统一并行语义（TP/EP）。

### 算法/流程文字说明
- **Bucket触发判定**：当所需槽位全部到达或标记为不可达时，Bucket触发前向计算。
- **activation id生成**：根据下游是否有Bucket，选择规则C（使用钻石根种子）或规则A（使用当前节点种子），哈希得到aid。
